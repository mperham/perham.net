Groovy vs JRuby

I spent some time playing with Groovy today. It’s somewhat similar to Ruby in that it is an interpreted language like Ruby but with tight Java integration (in fact Groovy natively executes within the Java VM by generating bytecode at runtime). Comparing its feature list, Groovy looks like a 100% rip off of Ruby (well, imitation is the …) but not implemented as professionally. Its founder, James Strachan, is well known in the Java community as quite gifted for starting visionary open source projects (e.g. ActiveMQ, Groovy, ServiceMix, et al) but not exactly polishing them to a gleam.

Now along comes something like JRuby. Ruby implemented on top of the Java VM, rather than a native interpreter. It still has yet to gain any sort of momentum but it has the draw of Ruby and Rails: a powerful language combined with an incredible webapp framework.

My guess is people who use Java will stick with Groovy, warts and all. It’s on the JCP fast track to become THE java scripting language so even though it may look like a pig with lipstick right now, it’ll eventually become something respectable and dependable in a production environment.

Those who aren’t constrained to a particular environment can evaluate Ruby. Quite frankly Ruby is just as universal as Java, given that it can run on Linux, Windows and Mac OSX. It’s not as polished, what with pathetic tool and IDE support, but it does have a certain clean slate appeal to it that Java, with its 100MB IDEs and monstrous enterprise application servers, cannot match. Unfortunately Ruby does not have the mature and vast array of frameworks out there to do everything like Java. Want to do text searching? Out of luck in Ruby, use Lucene in Java. Want a nice web UI framework? Ruby templates SUCK (they look identical to JSP 1.0, and that SUCKED too), Java has Tapestry. In a perfect world, changing platforms wouldn’t involve two steps forward, one step back. Alas.
