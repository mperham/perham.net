November 06, 2002

DB2 disappointment

I promised DB2 numbers and spent all morning getting it running with our system only to find that it has some really stupid limitations: index and constraint names are limited to 18 characters. This means that our system, which is packed with index names like IDX_QUESTION_QUESTIONID, will not work without significant reworking of our schema. I cannot believe that a modern high-performance database like DB2 has this pathetic limitation.

So I gave up. We will not be supporting DB2. So far we have only gotten Oracle, MSSQL and postgresql to run without incident.

November 05, 2002

Database Findings

My chosen profession is that of the software architect. I write large scale multi-tier systems that cost a lot of money. An integral part of these systems is the database. The relational database was invented in the early 70s with the first software examples appearing in the mid 80s or thereabouts.

As an architect, part of my job is providing as many choices as I can to the customer so they can tailor their installation of my software to suit their own needs/desires. One of the big choices is that of the database - which database should I choose? There are many choices:

Commercial:
Oracle
Sybase
Microsoft SQL Server
IBM DB2

Free:
postgresql
mysql
sapdb

When it comes right down to it, those 7 choices cover 99% of all databases in the world. Oracle is basically presumed to be the industry leader and having the fastest, most scalable offering. But this was all rumor - until today.

We've been pushing for database independence at work so I've been investigating most of the major offerings to see if they are potential candidates for use with our software. In doing so, I basically tried to get our unit test suite to run on each database. The platform is Windows XP with a 2.5Ghz P4. Here's the timing results of the test suite as run on each system:

Oracle - 3:00
SQL Server - 3:45
postgresql - 6:00
sapdb - unable to run our app

The unit tests very much test the speed of the database out of the box. I am not a DBA and none of the systems were tuned in any way. SAPDB has some serious limitations in how much data you can return in a SELECT statement (8k!) which basically makes it useless for large J2EE applications. BUT what I could get to run on SAPDB was blazingly fast. With SAPDB, our cleandb procedure, which deletes the entire tablespace and rebuilds it from a set of SQL files, took 1/3 of the time it took on Oracle. If you can live with its limitations, give it a try.

Overall I'm a little disappointed with postgresql. It was stable and relatively easy to get working with our system but the performance is nowhere near the speed of the commercial guys. I can't test Sybase because it only supports Windows NT right now (!). I'm not even going to try mysql simply because our customers would laugh us out of the room if we told them we support it. I'm going to try to get DB2 up and running tomorrow.

In closing, for the past 5 years I've always heard that Oracle was solid but not necessarily the fastest database. Now I understand why they charge so much for it - it's solid, fast and flexible.

November 02, 2002

Excuse me while I gripe...

How does Windows Media Player suck? Let me count the ways:


  • The interface is completely nonstandard and difficult to use.
  • It's crippled by draconian copy protection default settings.
  • Makes performing simple tasks far too difficult.

First off the interface is all flash and no substance. It throws out years of windows standards for pretty widgets that impress for about 30 seconds and then grow tiring. You can't use shortcut keys for the standard operations like play, forward, reverse, etc. Try changing your system fonts - now the interface is horrible because it was designed to look good with the default system settings only. Almost all common operations require the mouse to accomplish.

By default the program can only rip CDs to the Windows Media Audio format and copy protection is turned on. This means if you copy your files to another machine, they will not play there! If you reinstall your operating system, they will not even play on the same machine! The first thing I did when I started WMP was enable MP3 ripping and turned off all copy protection. It's my data that I paid $15 for - I'll decide where and how I want to listen to it!

Finally it makes simple tasks far too difficult. I literally could not figure out how to burn an Audio CD. You would think you would select "Create CD", pick the tracks you want to burn and hit Go. Nope. You can only burn tracks that are in your media library. Obviously this is another copy protection scheme as you are no doubt limited as to which files you can add to your library and how many times you can burn them. Once the files are imported, burning them to CD is relatively painless.

So my opinion is that WMP is a program designed to please the recording industry, not provide a solid, easy to use media program for the user. I'm going to stick with Winamp.
