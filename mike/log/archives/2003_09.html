September 22, 2003

Image Trickery

I played around with the Java Advanced Imaging API (JAI) a year ago when I was trying to come up with a solution to creating photo albums for my vacation pictures. I just wanted something to iterate through all pictures in a directory, create 16th sized thumbnails and an index page which can be used to view those thumbnails. I was able to create such a program, Genny, which I have used for a few of my picture albums but sometimes the thumbnails look terrible. I did some research and it turns out to just be bugs in the JAI. Nothing I can do about it without some massive debugging and learning imaging algorithms. As a side note, I really should have written Genny in a scripting language like Perl or Python - there's no need for the complexities of Java. At the time I had planned to put a Swing interface on top of it though...

I discovered JMagick today which is a Java wrapper around the great ImageMagick libraries. Supposedly much faster and much higher quality than the JAI algorithms. I've used ImageMagick in the past with great success so I'll have to add this to my toolbelt.

September 18, 2003

Ch-ch-ch-changes...

I'm constantly amazed at how fast and yet how slow the tech world seems to move. This time two years ago I was using BEA Weblogic, Oracle, Struts, and ant as the major components in building enterprise web applications. Now I'm using Websphere and resin, mysql and DB2, hibernate, velocity and tapestry, and maven. The world seems to move fast because I have to learn new systems constantly as new tools constantly become better than existing tools. It seems slow though because the tools are essentially interchangable. There is no huge revolutionary leap in any of these tools, just constant evolution.

One thing that is profoundly changing how development is done is the maturing world of open source tools. With very few exceptions, the most popular tools these days are built by volunteers at places like Apache and sourceforge. This is making our lives as developers radically easier because we can get the code for these projects in order to track down problems or customize for special requirements. 10 years ago, if you wanted an application component, you looked at the ads in development magazines for relevant tools.

When I wrote AppBar in 1996, I released the source for it on the web. It was literally the first open source application I had ever seen on Windows. At the time, Windows had a very strong freeware and shareware base and no precedent for open source; no one released their source code. I learned from the source code examples available in MSDN but I didn't want others to have as hard as a time (and MSDN cost $$$) and wanted to give other people the opportunity to customize AppBar to suit their needs. For my trouble I received $116 and a German translation of the AppBar interface in contributions. Was it worth it? Absolutely. :-) I'm proud to have been part of the push for openness in development.

September 10, 2003

Web Services Rant

Support for both document-passing and RPC-invocation semantics in web services is the worst decision the WSDL developers ever made. It makes interoperability between different systems a living hell for developers and will cripple web services from being deployed widely until the industry settles on a common, well understood format.

Currently there are three formats: rpc/encoded, rpc/literal and document/literal. WS-I has come out and said that SOAP encoding is terrible and should not be used which is true and rules out rpc/encoded. The problem is that pretty much ever system until now has defaulted to rpc/encoded so that is what most people use, including the public services exposed by Amazon and Google. Axis, the leading free and open source web service server does rpc/encoded very well and everything else pretty poorly.

But that's not enough. Even with rpc/literal, you do not have absolute control over the generated SOAP. Instead the SOAP framework generates small pieces of the SOAP document which can lead to interop problems. document/literal is the only way to get absolute control of the generated document and that is what WS-I should have mandated. This is fundamental architecture - giving people choices here just makes for problems and delays adoption.
