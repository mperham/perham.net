Scalable Browser-based Applications

February 19th, 2006 · 2 Comments

The Web Browser has been a phenomenally successful platform for content deliverly. HTML/CSS is a relatively simple method to create cross-platform user interfaces and the PULL model is great when you want to simply create a content repository which users can browse (e.g. www.cnn.com or any other news site).

Unfortunately the browser is not designed for applications which require a PUSH content delivery model. Consider group messging - how do you create an application where 4 people can chat together? Browsers talk to servers so you can’t link them directly to each other (ignoring the issue of firewalls). Instead you provide a central server and all 4 users connect to that server. This is traditionally how IM clients work. So User 1 types a message into his browser and hits return. The browser sends the message to the server. But how do the other three users get notified of the new message? The simple answer is they poll. They contact the server every X seconds asking if there are new messages. Effectively they are pulling new messages from the server rather than the server pushing new messages to them.
But this is not scalable - the client and the server are doing extra work when there are no new messages. So the obvious question becomes - “Is there a way to push content to the browser?”

As it turns out, there is but it requires some hacking. Google has been leading this effort - they recently integrated their Google Talk service with Gmail and so now it requires more resources. Instead of polling the mail server every 5 minutes for new mail, you need to poll the chat server every 3-5 seconds. So they discovered how to use the PUSH model in modern browsers. Incredible.

I discovered this while researching a new Ruby on Rails application, Campfire. If Rails can integrate this push model, it can truly become a framework for building novel browser-applications applications which were simply impossible a year ago.
